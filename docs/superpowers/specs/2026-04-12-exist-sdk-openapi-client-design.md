# exist-sdk OpenAPI Client — Design

**Date:** 2026-04-12
**Topic:** Generate an OpenAPI client from `docs/exist-api-openapi.yaml` and package it as an ESM module

---

## 1. Overview

A TypeScript SDK for the Exist.io read-only API surface, targeting both Node.js and Deno with ESM modules. The client is generated from the existing OpenAPI spec and wrapped with a thin fetch-based HTTP layer. Token management is external (caller provides the token).

## 2. Scope

### In Scope (Phase 1 — Read-Only)

| Endpoint                         | Method | Description                                        |
| -------------------------------- | ------ | -------------------------------------------------- |
| `/api/2/auth/simple-token/`      | POST   | Exchange username/password for a simple read token |
| `/api/2/accounts/profile/`       | GET    | Current user profile                               |
| `/api/2/attributes/with-values/` | GET    | Paged attributes with values                       |

### Deferred (Future Phases)

- Write endpoints: `POST /api/2/attributes/acquire/`, `/release/`, `/create/`, `/update/`, `/increment/`
- OAuth2 endpoints: `/oauth2/authorize`, `/oauth2/access_token/`
- Averages and Correlations endpoints

## 3. Architecture

```
exist-sdk/
├── src/
│   ├── client.ts           # Core fetch-based HTTP client
│   ├── types.ts            # Generated types from openapi-typescript
│   ├── endpoints/
│   │   ├── auth.ts         # Simple token exchange
│   │   ├── account.ts      # Profile
│   │   └── attributes.ts   # Attributes with values (read-only)
│   └── index.ts            # Public exports
├── scripts/
│   └── generate-types.ts   # openapi-typescript codegen script
├── package.json
├── tsconfig.json
└── tsx.res.json
```

### 3.1 Client (`src/client.ts`)

A minimal wrapper around `fetch`. Exports a single `createClient(opts)` function:

```typescript
interface ClientOptions {
  token: string;
  baseUrl?: string; // default: "https://exist.io/api/2/"
  fetch?: typeof fetch; // allows injection (undici for Node <18)
}

function createClient(opts: ClientOptions): ExistClient;
```

`ExistClient` exposes typed endpoint methods. It:

- Injects `Authorization: Token <token>` on every request
- Serializes request bodies as JSON
- Parses response bodies as typed JSON
- Throws `ExistError` on non-2xx responses

### 3.2 Types (`src/types.ts`)

Generated by `openapi-typescript` from `docs/exist-api-openapi.yaml`. Re-generated via `yarn generate-types`. Only the schemas used by in-scope endpoints are referenced; unused schemas are pruned by the tool.

### 3.3 Endpoints (`src/endpoints/`)

Each endpoint is a typed function that takes a `client` and request parameters, calls `client.request(path, options?)`, and returns the parsed response.

No class wrappers — pure function exports that compose with the client.

### 3.4 Runtime Compatibility

- **Deno:** Native `fetch` — no polyfill needed
- **Node.js 18+:** Native `fetch`
- **Node.js <18:** Caller can pass an `undici`-based `fetch` implementation via `opts.fetch`

## 4. Error Handling

All errors surface as `ExistError`:

```typescript
interface ExistError {
  status: number;
  message: string;
  code?: string;
}
```

| Scenario         | Behavior                                                      |
| ---------------- | ------------------------------------------------------------- |
| HTTP 4xx/5xx     | `ExistError` with `status` and `message`                      |
| Network failure  | `ExistError` with original `cause`                            |
| Rate limit (429) | `ExistError` with `status: 429` and `message: "Rate limited"` |

No automatic retry. No circuit breaker. Caller decides retry policy.

## 5. Package Scripts

```json
{
  "scripts": {
    "generate-types": "tsx scripts/generate-types.ts",
    "typecheck": "tsc --noEmit",
    "test": "vitest"
  }
}
```

`tsx` is used instead of `ts-node` for ESM compatibility with Yarn 4 PnP.

### 5.1 `generate-types.ts`

```typescript
// scripts/generate-types.ts
import { generateTypes } from "openapi-typescript";
import { writeFileSync } from "node:fs";

const spec = new URL("../docs/exist-api-openapi.yaml", import.meta.url);
const output = new URL("../src/types.ts", import.meta.url);

const types = await generateTypes(spec, { output });
writeFileSync(output, types);
console.log("Generated src/types.ts");
```

## 6. Testing

### Unit Tests

- Mock `fetch` at the client level — prove correct URL construction, header injection, body serialization
- Endpoint function tests pass typed args and verify the caller's `client.request` is invoked correctly

### Contract Tests

- Re-run `generate-types.ts` on spec changes and diff the output — catch drifted types before publish

## 7. Dependencies (Phase 1)

| Dependency           | Purpose                                      |
| -------------------- | -------------------------------------------- |
| `openapi-typescript` | Type generation from YAML spec               |
| `tsx`                | ESM-compatible TypeScript runner for scripts |
| `vitest`             | Unit testing                                 |

No runtime dependencies. `fetch` is provided by the host runtime.
