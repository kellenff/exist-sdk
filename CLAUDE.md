## SDK Pattern

All endpoint functions follow `(client: ExistClient, params?) => Promise<T>`:

```ts
const profile = await getProfile(client);
const attrs = await getAttributesWithValues(client, {days: 7, limit: 50});
```

Each endpoint is a standalone async function. No class instantiation. Easy to test, easy to tree-shake.

## Type Generation

`src/types.ts` is auto-generated by openapi-typescript from `openapi.yaml`. Do not edit manually — regenerate with `yarn generate-types`. Add re-exports in the `// Re-export commonly used types` section near the bottom.

## Endpoint Implementation

Each module has a local `buildQuery` helper and a `validate` helper:

```ts
function buildQuery(params: object): string {
  const entries = Object.entries(params).filter(([, v]) => v !== undefined);
  if (entries.length === 0) return '';
  return '?' + entries.map(([k, v]) => `${k}=${encodeURIComponent(String(v))}`).join('&');
}

function validate<T>(schema: z.ZodSchema<T>, data: unknown, errorMessage: string): T {
  const result = schema.safeParse(data);
  if (!result.success) {
    throw {status: 0, message: errorMessage, cause: result.error.issues} as ExistError;
  }
  return result.data;
}
```

## Key Architectural Decisions

**Branded token types** — `ApiToken` and `UserToken` are distinct Zod-branded string types. TypeScript treats them as incompatible. Use `ApiTokenSchema.parse()` to validate at runtime.

**Token store abstraction** — `TokenStore` interface in `src/oauth/token-store.ts` is the boundary. `FileTokenStore` uses a platform layer in `src/oauth/file-store.ts` to handle Node/Deno/Bun filesystem differences.

**OAuth2 via oauth4webapi** — Protocol complexity (OIDC discovery, PKCE, token refresh, device polling) is delegated. The SDK wraps it in a clean TypeScript interface.

**Zod validation on every response** — Every endpoint validates its response against a Zod schema. If the API shape changes, you get a typed `ExistError` with `cause` containing the specific schema failures.
